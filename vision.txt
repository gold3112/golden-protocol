核心
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
ブラウザは窓だった。次に来るのは箱だ。
インターネットに「入る」体験を作る。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
何が問題か
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
今のインターネットは部屋の集合体だ。サービスをまたぐたびに自分が分断され、
文脈が失われ、道がない。ブラウザという窓は30年間その構造を変えなかった。

今のウェブは「意図から始まる」——検索する、URLを打つ、ページが来る。
何もしなければ何も起きない。完全にpull-based。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
何を作るか
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
誰も所有しない共有空間——街。

  24時間眠らない。世界中の誰かが必ずいる
  アクセス手段を問わない。2Dでも3Dでも、CLIでも、同じ空間に入れる
  既存のインターネットと完全な互換性を持つ
  基盤はオープン。誰もが上にサービスを建てられる

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
体験の核心——「散策」
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
「歩く」ではなく「散策」。
目的のない移動、偶然の出会い。意図がなくても、存在しているだけで空間が応答する。

意図がなくていい。ただ、そこにいる。
  存在する → 空間が応答する → 奥行きに気づく → 引き寄せられる

ユーザーは何もしていない。なのに何かが起きている。
その驚きが最初の体験でなければならない。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
情報の奥行きとは何か
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
メタバースの失敗は「3Dで空間を描こうとした」ことにある。
奥行きは視覚的深度ではない——情報構造の問題だ。

「見えないが、ある」という感覚が奥行きの正体。
夜の都市の端に立ったとき、目の前の街灯と遠くの光の海だけが見える。
3Dモデルがなくても、まだ見ていない何かが無限にあると感じる。

CLIのテキスト画面でも3D空間でも、奥行きは実装できる。
なぜなら奥行きはレンダリングの問題ではなく、情報構造の問題だから。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
プロトコルが返すもの——「場の状態」
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
サーバーはページを返さない。場の状態を返す。

  position  : あなたが今いる場所（座標ではなく文脈）
  density   : この場所の情報密度
  presence  : 今ここにいる気配の数
  near      : 近くにあるもの（見える）
  horizon   : 存在はわかるが見えていないもの  ← 奥行きの実装
  drift     : 空間の流れ・傾き
  timestamp : 場は時間を持つ

nearとhorizonの対比が奥行きの正体になる。

UIはこの状態を受け取って、自分の形式で表現する。
CLIなら文字の濃淡と配置で。2Dなら光と影で。3Dなら物理空間で。
プロトコルが返すのは空間の状態。描き方はクライアントに委ねる。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
UXの原則——HTTPSのように隠す
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
HTTPSを誰も理解していないのに全員が使っている。
なぜなら存在を意識させないから。

新プロトコルも同じ構造でなければならない:
  ユーザーが見るもの    → 今までと同じ入口（または何もない）
  ユーザーが知ること    → 何もない
  実際に起きていること  → 場の状態が流れている

「ox://」のような新スキームを使わせてはいけない。
それは「知っている人だけが使う扉」になる——ダークウェブと同じ失敗構造。

新プロトコルは既存のDNS・HTTPの上に重なって流れる。
ユーザーは設定しない。空間がそこにある。
UXが大事。これは直接、生活の豊かさに直結する。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
技術的方向性
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
三つの新しい層を既存プロトコルの上に定義する:

  アイデンティティ層  — プラットフォームをまたいで持続する自分
  空間層            — URLを超えた、散策できる新しい参照系
  インタラクション層  — 既存サービスと高解像度で繋がる拡張仕様

Core Protocolの中核にLLMを置く:
  あなたの存在（identity + 履歴）
          ↓
     LLMが場を生成する
          ↓
  場の状態（density, near, horizon...）
          ↓
     UIが表現する（無限の表現形式）

場は静的データではなく、あなたが来たから生まれる。

設計方針:
  Core Protocolを先に作る（場の状態の定義・管理）
  UI Layerは後から何通りでも繋げられる
  今使える最大の技術で理想をそのまま実装する——妥協点を探さない
